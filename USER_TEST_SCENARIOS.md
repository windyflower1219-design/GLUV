# GLUV 사용자 테스트 시나리오 (UAT - User Acceptance Testing)

본 문서는 GLUV 애플리케이션의 서비스 품질을 보장하기 위해 실제 사용자의 시점에서 수행해야 할 테스트 기준과 시나리오를 정의합니다. 모바일 환경(PWA)에서의 구동 여부와 AI 기반 핵심 기능의 정상 동작을 중점적으로 살펴보기 위해 작성되었습니다.

## 1. 인증 및 가입 (Authentication)
**목표:** 사용자가 안전하게 계정을 생성하고 앱 내 기능에 접근할 수 있는지 확인합니다.

| 테스트 ID | 테스트 케이스 / 조건 | 예상 결과 |
| --- | --- | --- |
| AUTH-01 | 신규 이메일 가입 | 유효한 정보 입력 시 회원가입이 완료되고 `/dashboard`로 이동해야 함. |
| AUTH-02 | 기존 이메일 로그인 | 유효한 정보 입력 시 로그인에 성공하고 즉시 홈 대시보드 화면이 표출됨. |
| AUTH-03 | 잘못된 인증 정보 입력 제한 | 유효하지 않은 비밀번호나 존재하지 않는 계정 입력 시 명확한 에러 메시지가 노출되어야 함. |
| AUTH-04 | 비로그인 사용자 접근 차단 | 로그인하지 않은 상태에서 강제로 `/dashboard` 등 접근 시 방어 로직(Route Guard)에 의해 `/login`으로 리다이렉트됨. |

## 2. 홈 대시보드 (Dashboard)
**목표:** 사용자가 자신의 핵심 건강 요약 데이터를 첫 화면에서 명확히 인지할 수 있는지 확인합니다.

| 테스트 ID | 테스트 케이스 / 조건 | 예상 결과 |
| --- | --- | --- |
| DASH-01 | 최신 데이터 출력 여부 확인 | 금일의 최신 혈당 수치, 전체 섭취 칼로리 요약 내용이 StatCard 형식으로 명확히 표시됨. |
| DASH-02 | 혈당 게이지 UI(GlucoseGauge) 직관성 | 현재 혈당의 수치 정도(정상, 주의, 위험 등)를 시각적인 게이지와 컬러로 뚜렷하게 표현해야 함. |
| DASH-03 | 세부 메뉴 진입 | 대시보드 통계 카드 터치 시 대응하는 세부 메뉴(혈당 또는 식단 기록 페이지)로 정상 이동함. |

## 3. 음성 기반 통합 입력 (Voice Input)
**목표:** 주요 차별점인 음성 인식 기능이 원활하게 작동하고 유저 발화 내용이 올바른 수치 데이터로 분해되는지 확인합니다.

| 테스트 ID | 테스트 케이스 / 조건 | 예상 결과 |
| --- | --- | --- |
| VOC-01 | 마이크 권한 요청 및 UI 노출 | 음성 입력 시작 버튼 탭 시, 마이크 권한 안내 팝업이 노출되며 권한 허가 시 활성화 상태 표시가 됨. |
| VOC-02 | 음성-텍스트 인식 구동(Speech to Text) | "점심으로 샌드위치 먹었고, 식후 혈당은 115 나왔어" 발화 시, 해당 텍스트가 정확도 높게 모달창에 노출되어야 함. |
| VOC-03 | AI 데이터 파싱 (OpenAI) | 음성 텍스트 분석 완료 후, [식단 내역: 샌드위치], [혈당 수치: 115] 형태의 정형 데이터로 화면에 분리 표출됨. |
| VOC-04 | 파싱 데이터 수동 편집 (Edit) | AI가 생성한 음식명, 수량, 칼로리, 탄수화물 등의 내역을 유저가 인라인(Inline)으로 직접 터치하여 즉시 수정할 수 있어야 함. |
| VOC-05 | 시간적 맥락 지능형 파싱 (Time Context) | 예: 저녁 18:00에 "아침에 사과 먹었어"라고 입력할 경우, 현재 시각인 저녁이 아닌 아침(예: 08:00 등)으로 판단하여 기록 시각이 자동 세팅되어야 함. |
| VOC-06 | 저장 및 예외 처리 | 유저가 결과를 최종 확인 후 '저장' 버튼 터치 시 수정/반영된 상태로 Firestore에 지연 없이 기록됨. 인식 실패 시 재시도 안내 및 수동 입력 옵션 부여. |

## 4. 건강 기록 관리 (Glucose & Meals)
**목표:** 사용자가 직관적인 UI를 통하여 혈당이나 식단의 조회 및 수동 입력을 거침없이 수행할 수 있는지 검증합니다.

| 테스트 ID | 테스트 케이스 / 조건 | 예상 결과 |
| --- | --- | --- |
| REC-01 | 혈당 수동 입력 | 기록 시점(식전/식후 등)과 시간, 수치를 기입해 수동 저장을 시도하면 리스트에 즉각 반영되어야 함. |
| REC-02 | 식단 수동 입력 | 식사 종류와 예상 칼로리 등을 입력 후 정상적으로 식단 리스트에 추가되어야 함. |
| REC-03 | 데이터 로딩 및 보안 확인 | 타인의 UID로는 데이터를 읽어올 수 없으며 본인의 기록만 최신 순/시간 순으로 정확하게 조회됨. |

## 5. AI 맞춤형 인사이트 (Insights)
**목표:** 축적된 개인의 건강 데이터를 바탕으로 유의미한 AI 피드백을 적절한 시점에 제시하는지 확인합니다.

| 테스트 ID | 테스트 케이스 / 조건 | 예상 결과 |
| --- | --- | --- |
| AI-01 | 인사이트 생성 및 표출 | 최근 입력된 식단 및 평균 혈당 데이터가 있을 때, `/insights` 진입 시 Gemma 4를 통해 분석결과가 생성/표출됨. |
| AI-02 | 유저 상태별 맞춤형 톤 앤 매너 | 혈당 위기 발생 시에는 경고형 조언이, 상태 호전 시에는 긍정/공감형 코멘트가 노출되는 등 데이터 상황에 부합하는 내용이 생성됨. |

## 6. 사용성 및 PWA 모바일 경험 (Usability & PWA)
**목표:** 작은 모바일 화면부터 PWA 설치 환경까지 네이티브 앱 수준의 사용 경험을 제공하는지 확인합니다.

| 테스트 ID | 테스트 케이스 / 조건 | 예상 결과 |
| --- | --- | --- |
| UX-01 | 하단 바 내비게이션 부드러움 | BottomNavigation을 통한 메뉴 전환 시 UI 깜빡임이 없고 현재 체류 중인 메뉴가 강조 표시되어야 함. |
| UX-02 | 모바일 구조 (SafeArea 해상도 최적화) | 스마트폰 기기의 노치, 펀치홀, 하단 스와이프 바 등에 의해 버튼이나 텍스트가 가려지는 영역이 없어야 함. |
| UX-03 | 홈 화면 추가 기능 (PWA 설치) | 모바일 브라우저(Safari, Chrome)의 '홈 화면에 추가'를 통해 앱을 설치하고, 실행 시 풀 스크린 독립 앱처럼 런칭됨. |
| UX-04 | 네트워크 단절 시(오프라인) 대처 조치 | 비행기 모드 등 오프라인 환경에 들어갔을 때, Service Worker의 캐시로 레이아웃이 렌더링되며 통신 불가 상태 안내가 출력됨. |

## 7. 성능 및 응답 속도 (Performance & Latency)
**목표:** 외부 서버 및 AI 모듈(OpenAI, Gemma 4), 데이터베이스(Firestore) 통신 과정에서 사용자가 체감하는 대기 시간을 최소화하고, 수용 가능한 기준 내에서 동작하는지 확인합니다.

| 테스트 ID | 테스트 케이스 / 조건 | 예상 결과 |
| --- | --- | --- |
| PERF-01 | 앱 초기 로딩 속도 | 브라우저/PWA에서 앱 실행 시, **2초 이내**에 초기 대시보드 화면 렌더링이 완료되어야 함. |
| PERF-02 | 음성-텍스트 변환 딜레이 | 음성 발화를 종료한 시점으로부터 모달창에 텍스트가 표시되기까지 지연이 **1초 미만**이어야 함. |
| PERF-03 | AI 데이터 파싱 소요 시간 | 음성 텍스트 확인 후 '저장/분석' 요청 시, OpenAI API 연동 및 정형 데이터 표출까지 **3초 이내**에 완료되어야 함. (작업 중 스피너 등 로딩 인디케이터 노출 필수) |
| PERF-04 | 데이터베이스(Firestore) 반영 속도 | 데이터를 기록(저장)할 때, **최대 1~1.5초 이내**에 기록이 완료되고 현재 보고 있는 페이지 목록에 즉시 갱신(반영)되어야 함. |
| PERF-05 | 맞춤형 인사이트 생성 소요 시간 | `/insights` 탭 진입 후, Gemma 4 AI의 응답 데이터가 **최대 3~5초 이내**에 노출되어야 함. (대기 시간 체감 완화를 위한 스켈레톤 UI 표출 필수) |
